# 二次释放审计清单 D1-D8 (必须逐条仔细检查)

## D1 同函数内重复释放

同一变量在同一执行路径上出现两次释放调用即为 bug。不同互斥分支各释放一次是正常的。

## D2 正常路径 + 错误路径双重释放

资源在正常路径已释放，ereport(ERROR) 后 PG_CATCH 中又释放。也要检查 PG_TRY/PG_CATCH 都调用的清理函数是否幂等（中途 ERROR 后重入会导致已释放资源二次操作）。

对 PG_TRY 中每个释放点，必须检查该释放点**之后到 PG_TRY 结束**之间的所有 ERROR 触发点，不能只检查释放操作本身能否失败。

## D3 调用者-被调用者双重释放

函数 F 内部已释放资源，检查 F 的所有调用者是否在调用 F 后又释放同一资源。重点关注 CleanupXXX 类清理函数。

## D4 释放后未置空

`pfree(ptr)` 后未置 `ptr = NULL`，追踪该指针后续是否可能被再次释放。

## D5 所有权转移后残留释放

资源通过赋值或入队转移给其他结构体/函数后，原持有者不应再释放。检查：
- `struct->ptr = field` 后 field 是否仍被释放
- 入队/入池操作后是否仍调用释放函数（业务扩展中列有具体转移函数）

## D6 结构体字段与局部变量双重释放

`ptr = struct->field` 后，`pfree(ptr)` 与 `pfree(struct->field)` 释放同一块内存。

## D7 锁的重复释放

同一路径上 `LWLockRelease` / `SpinLockRelease` 出现两次，或子函数已释放锁、父函数又释放一次。

## D8 与对等函数对比释放次数

找同模块中功能相似的对等函数，对比各自的释放调用数量。数量不一致时，多的一侧存在重复释放风险（业务扩展中列有具体对等文件对）。
